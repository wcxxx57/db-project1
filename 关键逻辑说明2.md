# 关键逻辑说明（第二阶段）

本文档是第二阶段系统的关键业务逻辑总览。第二阶段的核心变更是将题目从问卷内嵌结构拆分为独立的、带版本管理的 `questions` 集合，问卷通过引用 (`question_ref_id` + `version_number`) 关联题目，由此衍生出版本管理、共享、题库、跨问卷统计等一系列新机制。

---

## 1. 题目版本管理逻辑

题目版本管理是第二阶段的核心机制。每道题目拥有一条不可变的版本链，所有变更都以追加新版本的方式记录，历史永远不被覆盖。

### 1.1 数据结构

`questions` 集合中每个文档的核心结构如下：

```javascript
{
  "_id": ObjectId("..."),            // 题目谱系 ID（question_ref_id）
  "latest_version_number": 3,        // 当前最新版本号
  "access_control": {
    "creator": "user_id",
    "shared_with": ["user_id_2"],
    "banked_by": ["user_id_3"]
  },
  "versions": [
    {
      "version_number": 1,
      "created_at": ISODate("..."),
      "updated_by": "user_id",
      "parent_version_number": null,   // v1 没有父版本
      "type": "single_choice",
      "title": "您的年龄段？",
      "required": true,
      "options": [...],
      "validation": {...}
    },
    // ... 后续版本追加到数组末尾
  ]
}
```

### 1.2 创建题目自动生成 v1

调用 `create_question()` 时，系统自动构建版本号为 1 的初始版本对象，`parent_version_number` 设为 `None`，`latest_version_number` 初始化为 1，整个文档通过 `insert_one` 一次性写入：

```python
question_doc = {
    "latest_version_number": 1,
    "access_control": {"creator": user_id, "shared_with": [], "banked_by": []},
    "versions": [version_obj],   # version_obj.version_number == 1
}
result = db.questions.insert_one(question_doc)
```

调用方无需手动指定版本号，系统保证第一个版本始终为 v1。

### 1.3 修改题目一律创建新版本，不原地覆盖

`create_new_version()` 是修改题目的唯一入口。无论是修改标题、更换选项还是调整校验规则，一律在 `versions` 数组末尾追加新版本文档，绝不修改已有版本的任何字段。

版本号的递增通过 MongoDB 的 `$inc` 原子操作完成，配合 `$push` 将新版本对象追加到 `versions` 数组，两个操作在同一条 `update_one` 中执行：

```python
db.questions.update_one(
    {"_id": ObjectId(question_id)},
    {
        "$inc": {"latest_version_number": 1},
        "$push": {"versions": version_obj},
    },
)
```

**`$inc` 原子递增保证并发安全：** 即使两个用户同时对同一题目创建新版本，MongoDB 在文档级别保证 `$inc` 的原子性，不会出现两个版本拿到相同版本号的情况。先到的请求将 `latest_version_number` 从 N 递增到 N+1，后到的请求在 N+1 的基础上递增到 N+2，版本号严格单调递增。

### 1.4 恢复旧版本时基于目标旧版本内容创建新版本

`restore_version()` 并不"回滚"——它读取目标旧版本的内容（`type`、`title`、`required`、`options`、`validation`），然后调用 `create_new_version()` 基于这些内容创建一个全新的版本，`parent_version_number` 指向被恢复的那个旧版本号：

```python
def restore_version(question_id, user_id, target_version_number):
    # ... 找到 target_version ...
    return create_new_version(question_id, user_id, {
        "type": target_version["type"],
        "title": target_version["title"],
        "required": target_version.get("required", True),
        "options": target_version.get("options"),
        "validation": target_version.get("validation"),
        "parent_version_number": target_version_number,
    })
```

这意味着恢复操作的结果是版本链上多了一个新节点，而非时间旅行式地删除后续版本。已引用旧版本的问卷不受任何影响。

### 1.5 已写入的历史版本不可修改、不可删除

系统中不存在任何修改 `versions` 数组内既有元素的 API 或内部方法。所有写操作只有两种：`$push`（追加新版本）和 `$inc`（递增版本号）。既没有 `$set` 覆盖某个版本的字段，也没有 `$pull` 从数组中移除版本。这保证了：

- 引用了某个 `(question_ref_id, version_number)` 的问卷和答卷，其数据永远可溯源。
- 统计分析永远能找到对应版本的题目内容，不会因版本被删而产生空引用。

## 2. 共享规则

题目的访问控制通过 `access_control` 子文档实现，包含 `creator`、`shared_with`、`banked_by` 三个字段。

### 2.1 仅创建者可彻底删除题目

`delete_question()` 首先校验 `access_control.creator == user_id`，非创建者直接被拒绝（403）。

即使是创建者，删除前还需检查是否有已发布（`status == "published"`）的问卷正在使用该题目的任何版本：

```python
published_surveys = db.surveys.find({
    "questions.question_ref_id": question_id,
    "status": "published",
})
if any(True for _ in published_surveys):
    raise QuestionServiceError(ErrorCodes.QUESTION_IN_USE,
        "该题目有已发布的问卷正在使用，无法删除", 400)
```

只有没有已发布问卷引用时，才允许执行 `delete_one`。此约束保护了正在收集中的问卷不会因题目被删而崩溃。

### 2.2 共享对象的权限范围

被共享者（`shared_with` 列表中的用户）可以执行以下操作：

- **查看题目详情及所有版本：** `get_question_detail()` 中 `_check_access()` 同时检查 `creator` 和 `shared_with`。
- **引用到自己的问卷：** `_validate_question_refs()` 校验引用合法性时，同样允许创建者或被共享者使用。
- **加入自己的题库：** `add_to_bank()` 经过 `_check_access()` 校验后，将 `user_id` 加入 `banked_by`。
- **创建新版本：** `create_new_version()` 经过 `_check_access()` 校验后，被共享者也可以为题目追加新版本。

被共享者**不能**执行的操作：
- 不能共享给其他人（`share_question()` 仅允许 `creator`）。
- 不能取消共享（`unshare_question()` 仅允许 `creator`）。
- 不能删除题目（`delete_question()` 仅允许 `creator`）。

### 2.3 按用户名定向共享

`share_question()` 接收目标用户的 `username`（而非 `user_id`），先查询 `db.users` 确认该用户名存在，再将其 `_id` 通过 `$addToSet` 加入 `shared_with` 数组。`$addToSet` 保证了重复共享不会产生重复条目。

取消共享时使用 `$pull` 从数组中移除目标用户 ID。

系统还阻止了自己共享给自己的操作（`target_user_id == user_id` 时返回错误）。

## 3. 问卷编辑规则（第二阶段变更）

### 3.1 问卷状态机与编辑约束

问卷有三种状态，编辑权限按状态区分：

```
draft (草稿) → published (已发布) → closed (已关闭)
     ↓              ↓                    ↓
  全量可编辑      禁止任何编辑         可编辑（有限）
```

`update_survey()` 中的状态检查：

```python
if survey["status"] == "published":
    raise SurveyServiceError(
        ErrorCodes.NO_PERMISSION,
        "问卷收集中不可编辑，请先关闭问卷",
        403,
    )
```

### 3.2 草稿问卷：完全开放编辑

草稿状态下，`update_survey()` 允许修改所有字段，包括：
- 元信息（标题、描述、截止时间、设置）
- 题目引用列表（增删题目、调整顺序、替换引用版本）

### 3.3 发布后：题目结构冻结

问卷处于 `published` 状态时，`update_survey()` 直接拒绝所有编辑请求。这保证了正在收集中的问卷结构稳定，不会出现：
- 正在填写的用户看到题目突然变化
- 已提交的答卷与新结构不匹配
- 跳转逻辑因目标题目被移除而失效

### 3.4 关闭后：允许编辑元信息和替换题目版本

问卷关闭后不再接受新答卷，可以安全地进行以下操作：
- 修改元信息（标题、描述、截止时间、设置）
- 替换题目引用的版本号（例如将某题从 v1 升级到 v2）
- 增删题目引用、调整顺序

### 3.5 questions 数组从内嵌改为引用

第一阶段中，问卷的 `questions` 数组内嵌了题目的全部内容（`type`、`title`、`options` 等）。第二阶段改为引用模式，每个数组元素的结构为 `SurveyQuestionRef`：

```python
class SurveyQuestionRef(BaseModel):
    question_id: str      # 问卷内部局部题号（如 q1, q2），供跳转逻辑路由使用
    order: int            # 题目在此问卷中的顺序
    logic: Optional[QuestionLogic]  # 跳转逻辑（属于该份问卷，不属于题目本身）
    question_ref_id: str  # 引用 questions._id，题目谱系 ID
    version_number: int   # 引用的具体版本号
```

题目的实际内容（`type`、`title`、`options`、`validation`）不再存储在问卷中，需要时通过 `_resolve_question_refs()` 从 `questions` 集合中按 `(question_ref_id, version_number)` 解析获取。

### 3.6 保存时校验引用合法性和用户权限

`update_survey()` 在写入 `questions` 数组前，依次执行两项校验：

**引用合法性校验 (`_validate_question_refs()`)：**

1. 批量收集所有 `question_ref_id`，使用 `$in` 一次性查询 `questions` 集合。
2. 逐条检查：
   - `question_ref_id` 和 `version_number` 都不能为空
   - 引用的题目文档必须存在
   - 当前用户必须是该题目的创建者或被共享者
   - 引用的版本号必须在该题目的 `versions` 数组中存在

**跳转逻辑校验 (`_validate_jump_logic_refs()`)：**

（详见第 4 节）

任何一项校验失败，都会拒绝整个保存请求，返回具体的错误信息。

## 4. 跳转逻辑引擎（保留第一阶段）

跳转逻辑引擎的核心算法在第二阶段中完全保留，只是运行在引用格式解析后的题目数据上。

### 4.1 三种条件类型

| 条件类型 | 适用题型 | 判定逻辑 |
|---------|---------|---------|
| `select_option` | 单选题 | `answer == condition.option_id` |
| `contains_option` | 多选题 | `any`模式：答案包含任一指定选项；`all`模式：答案包含全部指定选项 |
| `number_compare` | 数字题 | 支持 7 种运算符：`eq`/`ne`/`gt`/`gte`/`lt`/`lte`/`between` |

每道题目可配置多条跳转规则，引擎按规则数组顺序逐条评估，**首条命中的规则立即生效**（First-Match Priority），后续规则不再评估。

执行动作有两种：
- `jump_to`：跳转到指定的 `target_question_id`（问卷内部局部题号）
- `end_survey`：强制结束问卷

### 4.2 compute_required_questions 算法

该算法在答卷提交时计算用户实际需要回答的题目列表：

```
输入：questions（按 order 排序的题目列表）、answers_map（用户的答案映射）
输出：required_qids（用户实际需要回答的题目 ID 列表）

1. 从第一题（index=0）开始
2. 将当前题目 ID 加入 required_qids
3. 取出用户对当前题目的答案，调用 compute_jump_target() 评估跳转
4. 若返回 "__END__"：终止循环
5. 若返回某个 target_question_id 且该目标 index > 当前 index：跳转到目标
6. 否则：顺序前进到下一题
7. 重复步骤 2-6 直到所有题目处理完毕
```

**关键特性：** 只有 `required_qids` 列表内的题目才参与必填校验和答案保存。用户提交的跳过题目的答案会被静默丢弃，防止篡改。

### 4.3 跳转验证

问卷保存时，`_validate_jump_logic_refs()` 执行三层防护：

**第一层：目标存在性校验**
构建问卷内部 `question_id` 集合，确保 `jump_to` 的 `target_question_id` 必须存在于当前问卷的题目列表中。

**第二层：禁止向前跳转**
构建 `question_id → order` 映射表，确保跳转目标的 `order` 必须严格大于当前题目的 `order`。从根本上阻断回退跳转。

**第三层：循环检测**
使用深度优先搜索（DFS）遍历跳转有向图，检测是否存在环路。每次递归调用时传递 `visited.copy()` 保证路径独立性。即使是间接循环（Q1 -> Q3 -> Q5 -> Q3）也会被识别并拒绝保存。

## 5. 答卷提交逻辑（第二阶段变更）

第二阶段的答卷提交流程在校验前增加了题目引用解析步骤，写入时补齐引用信息。

### 5.1 提交前先解析问卷引用的题目版本

`submit_response()` 在校验答案之前，先调用 `_resolve_survey_questions()` 将问卷的引用格式题目解析为包含完整内容的题目列表：

```python
resolved_questions = _resolve_survey_questions(survey)
```

解析过程：
1. 收集问卷中所有 `question_ref_id`，去重后使用 `$in` 批量查询 `questions` 集合
2. 对每个引用项，在对应题目文档的 `versions` 数组中找到匹配 `version_number` 的版本
3. 将版本内容（`type`、`title`、`required`、`options`、`validation`）合并到引用项上

### 5.2 校验基于解析后的题目内容

解析完成后，校验流程与第一阶段一致，但数据来源从问卷内嵌改为引用解析：

1. `compute_required_questions(resolved_questions, answers_map)`：根据解析后的题目列表和跳转逻辑计算实际需要回答的题目
2. 必填校验：遍历 `required_qids`，检查 `required == True` 的题目是否都有答案
3. 答案值校验：调用 `validate_single_answer(question, answer)` 逐题校验，包括选项合法性、数值范围、文本长度等

### 5.3 写入时补齐 question_ref_id 和 version_number

构建答卷文档时，系统为每条答案补齐引用信息：

```python
for a in answers:
    if a["question_id"] not in required_set:
        continue   # 丢弃被跳过题目的答案
    q = question_map.get(a["question_id"])
    answer_doc = {
        "question_id": a["question_id"],
        "answer": a["answer"],
    }
    if q:
        answer_doc["question_ref_id"] = q.get("question_ref_id")
        answer_doc["version_number"] = q.get("version_number")
    answer_docs.append(answer_doc)
```

这样写入后的答卷文档中，每条答案都记录了它对应的题目谱系 ID 和版本号，使得后续的跨问卷统计可以直接按 `question_ref_id` 聚合，无需再回查问卷定义。

## 6. 统计逻辑（第二阶段变更）

### 6.1 按问卷统计基于解析后题目版本

`get_survey_statistics()` 首先调用 `_resolve_survey_questions()` 解析问卷中所有题目引用，获取每道题在其引用版本下的完整内容（标题、类型、选项列表等），然后基于这些内容构建统计结果。

这意味着统计结果始终反映问卷当前引用的题目版本内容。如果问卷关闭后将某题从 v1 升级到 v2（修改了选项文本），统计展示会使用 v2 的选项文本，但历史答卷中的 `option_id` 如果在 v2 中不存在，会被计入孤儿数据并生成 `warning` 字段提示。

### 6.2 跨问卷单题统计

`get_cross_survey_question_statistics()` 是第二阶段新增的统计能力，实现了按题目谱系跨问卷聚合：

**按 question_ref_id 聚合：**
1. 在 `responses` 集合中查找所有 `answers.question_ref_id == target_ref_id` 的答卷
2. 收集涉及的问卷 ID，批量查询 `surveys` 集合获取问卷信息

**按 version_number 分组：**
```python
for resp in all_responses:
    for ans in resp.get("answers", []):
        if ans.get("question_ref_id") == question_ref_id:
            vn = ans.get("version_number", 1)
            version_answers[vn].append(ans.get("answer"))
            version_survey_ids[vn].add(str(resp["survey_id"]))
```

每个版本分组独立统计，因为不同版本的题目内容（选项列表、题目类型）可能不同，混合统计没有意义。最终返回结构包含：
- 题目谱系信息（`question_ref_id`、最新版本标题/类型）
- 总回答数和涉及问卷数
- 各版本的分组统计（选项分布/文本列表/数字统计）
- 命中的问卷列表

**索引支持：** `responses` 集合上建有 `answers.question_ref_id` 索引（见 `database.py` 的 `init_indexes()`），保证跨问卷查询的性能。

## 7. 题库实现

### 7.1 banked_by 字段实现个人收藏夹

题库功能没有引入独立的 collection，而是复用 `questions` 集合中 `access_control.banked_by` 数组字段。该数组存储了所有将该题目加入题库的用户 ID。

**加入题库：**
```python
db.questions.update_one(
    {"_id": ObjectId(question_id)},
    {"$addToSet": {"access_control.banked_by": user_id}},
)
```

**移出题库：**
```python
db.questions.update_one(
    {"_id": ObjectId(question_id)},
    {"$pull": {"access_control.banked_by": user_id}},
)
```

**查询我的题库：**
```python
cursor = db.questions.find({"access_control.banked_by": user_id})
```

### 7.2 无独立 collection 的设计考量

采用 `banked_by` 数组而非独立的 `question_banks` collection，原因如下：

- **查询简单：** 一条查询即可获取用户的题库列表，无需 `$lookup` 或多次查询。
- **数据一致性：** 题目被删除时，`banked_by` 信息随文档自动消失，无需维护外键引用。
- **权限复用：** 加入题库前的权限校验（`_check_access()`）与查看题目的权限校验完全一致，无需额外的权限模型。
- **索引支持：** `access_control.banked_by` 上已建立索引，查询性能有保障。

加入题库的前提是用户对该题目有访问权限（创建者或被共享者），由 `_check_access()` 统一校验。

## 8. 数据迁移逻辑

迁移脚本 `backend/scripts/migrate_phase2.py` 实现了从第一阶段内嵌格式到第二阶段引用格式的一次性数据迁移。

### 8.1 从 surveys.questions 抽取题目写入 questions 集合

迁移脚本遍历所有问卷，对每个问卷的内嵌 `questions` 数组中的每个题目：

1. 提取题目内容（`type`、`title`、`required`、`options`、`validation`）
2. 构建 `questions` 集合的文档，包含 v1 版本和访问控制信息
3. `insert_one` 写入 `questions` 集合，获取生成的 `_id` 作为 `question_ref_id`
4. 建立 `question_id -> question_ref_id` 的映射关系

```python
for q in questions:
    question_doc = {
        "latest_version_number": 1,
        "access_control": {
            "creator": creator_id,
            "shared_with": [],
            "banked_by": [],
        },
        "versions": [version_obj],  # 包含原始题目内容的 v1
    }
    result = db.questions.insert_one(question_doc)
    qid_to_ref[q["question_id"]] = str(result.inserted_id)
```

### 8.2 回填引用字段

**回填问卷：** 将每个问卷的 `questions` 数组从内嵌格式改写为引用格式，保留 `question_id`（局部题号）、`order`、`logic`（跳转逻辑），新增 `question_ref_id` 和 `version_number`（固定为 1）：

```python
new_q = {
    "question_id": qid,
    "order": q.get("order", 0),
    "logic": q.get("logic"),
    "question_ref_id": ref_id,
    "version_number": 1,
}
```

**回填答卷：** 遍历该问卷的所有答卷，为每条 `answer` 补齐 `question_ref_id` 和 `version_number`（固定为 1），使得历史答卷也能参与跨问卷统计：

```python
for ans in answers:
    updated_ans = dict(ans)
    updated_ans["question_ref_id"] = qid_to_ref.get(ans["question_id"])
    updated_ans["version_number"] = 1
```

### 8.3 幂等性保护

脚本通过检查第一个题目是否已有 `question_ref_id` 来判断问卷是否已迁移过：

```python
if first_q.get("question_ref_id"):
    print(f"  问卷 {survey_id} 已经迁移过，跳过")
    continue
```

这保证了脚本可以安全地多次执行而不会产生重复数据。
