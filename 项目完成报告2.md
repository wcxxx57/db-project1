# 第二阶段项目完成报告

## 1. 第一阶段的设计是什么

第一阶段系统采用前后端分离架构（React + FastAPI + MongoDB），数据库使用三个核心集合：`users`、`surveys`、`responses`。

### 1.1 三集合结构

- **users 集合**：存储用户账户信息，包含 `_id`、`username`（唯一索引）、`password_hash`、`created_at`。
- **surveys 集合**：作为聚合根，内嵌问卷配置及完整的题目结构。包含 `_id`、`title`、`creator_id`、`access_code`（唯一索引）、`status`（draft/published/closed）、`response_count`（冗余计数）、`settings`（匿名/重复提交控制）、`questions`（内嵌题目数组）。
- **responses 集合**：记录完整作答快照。包含 `_id`、`survey_id`、`respondent_id`、`is_anonymous`、`submitted_at`、`answers`（内嵌答案数组）、`completion_time`。

### 1.2 surveys.questions 直接内嵌完整题目内容

问卷的 `questions` 数组直接内嵌每道题目的完整定义，包括题目类型（`type`）、题干（`title`）、选项（`options`）、校验规则（`validation`）、跳转逻辑（`logic`）等。一次查询即可获取问卷及所有题目，避免 JOIN 操作。

### 1.3 题目只属于单个问卷，无独立实体

题目作为问卷文档的嵌入子文档存在，没有独立的集合或 ID 体系。题目的生命周期完全依附于问卷——创建问卷时创建题目，删除问卷时题目随之消失。

### 1.4 支持的核心功能

- **跳转逻辑**：数据驱动的条件跳转引擎，支持 `select_option`、`contains_option`、`number_compare` 三种条件类型和 `jump_to`、`end_survey` 两种动作类型。保存时执行三层校验（目标存在性、禁止向前跳转、DFS 环路检测）。
- **答案校验**：参数化校验引擎，支持多选题选择数量限制、文本题字数限制、数字题范围和整数限制。采用双阶段校验管线（保存期配置自查 + 提交期运行时校验）。
- **匿名/重复提交控制**：通过 `settings.allow_anonymous` 和 `settings.allow_multiple` 控制，匿名提交时 `respondent_id` 设为 null。
- **统计**：按问卷维度聚合 `responses`，返回分题统计结果。

## 2. 新需求带来了哪些问题

第二阶段用户反馈了八项新需求（详见 `需求变更.md`），总结起来是希望题目能独立管理、复用、共享、版本化、跨问卷统计。这些需求暴露了第一阶段设计的以下问题：

### 2.1 题目无法复用（每次重新创建）

由于题目直接内嵌在问卷中，用户创建类似问卷时必须重新编写相同的题目。例如"你的年龄"这道题要在多个问卷中使用，每次都要重新输入，无法从已有题目中选择。

### 2.2 无法共享题目给团队

团队成员之间无法共享题目。如果 A 用户创建了一道好题，B 用户想在自己的问卷中使用，只能手动复制粘贴，无法通过系统机制实现共享。

### 2.3 修改题目影响已发布问卷

在内嵌模式下，题目内容直接存储在问卷文档中。虽然第一阶段对已发布问卷禁止了题目修改，但如果需要在不同问卷中使用"同一道题的不同版本"，内嵌模式完全无法支持——因为每个问卷里的"同一道题"实际上是独立的副本，没有任何关联。

### 2.4 无版本历史和追溯

题目没有版本概念，用户修改题目后无法查看历史版本、无法知道当前题目从哪个版本改来、无法恢复旧版本。

### 2.5 无法跨问卷统计同一道题

由于同一道题在不同问卷中是完全独立的副本，系统无法识别"这些是同一道题"，也就无法聚合所有问卷中该题的回答数据。例如"你的年龄"在 10 个问卷中使用，无法查看所有问卷中年龄的总体分布。

### 2.6 无题库管理

用户无法将常用题目整理成个人题库，也无法从题库中快速选题创建问卷。

## 3. 为什么需要修改原设计

### 3.1 内嵌模式的根本局限

第一阶段采用的"题目嵌入问卷"模式，其核心假设是"题目只属于单个问卷"。这个假设在新需求下被打破：

- **题目复用**要求同一道题能出现在多个问卷中，内嵌模式下只能复制副本，无法保持关联。
- **题目共享**要求多个用户引用同一个题目实体，内嵌模式下每个问卷的题目是独立的，修改一个不会影响另一个，也无法建立"共享"关系。
- **版本管理**要求题目有独立的版本链，内嵌模式下题目没有独立身份，无从谈起版本。
- **跨问卷统计**要求能按"同一道题"聚合，内嵌模式下不同问卷中的同名题目没有共同标识。

### 3.2 需要引入独立题目实体

要满足上述需求，必须将题目从问卷的嵌入子文档提升为独立实体，拥有自己的集合、ID、版本链和访问控制。问卷中的题目从"包含完整内容"改为"引用题目 ID + 版本号"。

这是一个数据模型层面的结构性变更，不是简单的功能添加，因此需要对原设计进行修改。

## 4. 修改了哪些部分

### 4.1 数据层：新增 questions 集合

新增独立的 `questions` 集合，每个文档代表一个题目谱系，内部使用 `versions` 数组保存全部历史版本，顶层维护 `latest_version_number` 和 `access_control`（创建者、共享列表、题库收藏列表）。

### 4.2 数据层：surveys.questions 改为引用格式

问卷的 `questions` 数组不再包含完整题目内容，改为引用格式：每个题目项仅保存 `question_id`（问卷内部局部题号）、`order`（顺序）、`logic`（跳转逻辑）、`question_ref_id`（引用 questions._id）、`version_number`（引用版本号）。

### 4.3 数据层：responses.answers 增加引用字段

答卷的 `answers` 数组中每个答案新增 `question_ref_id` 和 `version_number` 字段，支持跨问卷统计时按题目谱系和版本聚合。

### 4.4 后端：新增题目管理服务和路由

- 新增 `backend/app/services/question_service.py`：实现创建题目、查询列表（我的/共享/题库）、查看详情、创建新版本、查看版本历史、恢复旧版本、共享/取消共享、加入/移出题库、查看使用情况、删除题目等完整的题目域业务逻辑。
- 新增 `backend/app/routes/questions.py`：注册题目管理相关的所有 API 端点。

### 4.5 后端：问卷服务改造支持引用解析

重构 `backend/app/services/survey_service.py`：
- 问卷更新接口接收"题目引用列表"而非完整题目定义。
- 保存时校验 `question_ref_id + version_number` 是否存在且当前用户有使用权限。
- 读取问卷时批量解析引用，补全题目版本内容返回前端。
- 发布前校验基于"已解析的题目版本内容 + 问卷局部 logic"执行。

### 4.6 后端：答卷服务改造支持引用解析

重构 `backend/app/services/response_service.py`：
- 提交答卷前先解析问卷引用的题目版本。
- 校验、跳转、必填判断全部基于解析后的题目版本内容。
- 写入答卷时自动补齐 `question_ref_id`、`version_number`。

### 4.7 后端：统计服务新增跨问卷统计

重构 `backend/app/services/statistics_service.py`：
- 按问卷统计时，统计对象改为解析后的题目版本。
- 新增跨问卷单题统计接口：按 `question_ref_id` 聚合所有问卷的回答，按 `version_number` 分组展示。

### 4.8 前端：新增题目管理界面，编辑器改造

- 新增 `frontend/src/components/QuestionManager.tsx`：完整的题目管理界面，支持"我的题目"、"共享给我的题目"、"我的题库"三个标签页，包含新建题目、查看详情、创建新版本、恢复版本、共享管理、题库管理、使用情况查看、跨问卷统计等功能。
- 重构 `frontend/src/components/SurveyEditor.tsx`：编辑器题目列表改为维护"引用格式"，支持从题库/共享题目中选题加入问卷，支持查看引用版本号和替换版本。
- 修改 `frontend/src/components/StatisticsView.tsx`：增加跨问卷统计视图入口。
- 修改 `frontend/src/types/index.ts` 和 `frontend/src/services/api.ts`：新增题目相关类型定义和 API 调用函数。

## 5. 数据结构如何变化

### 5.1 新增 questions 集合（内嵌版本模式）

```json
{
  "_id": ObjectId,
  "latest_version_number": Number,
  "access_control": {
    "creator": String,
    "shared_with": [String],
    "banked_by": [String]
  },
  "versions": [
    {
      "version_number": Number,
      "created_at": DateTime,
      "updated_by": String,
      "parent_version_number": Number / null,
      "type": String,
      "title": String,
      "required": Boolean,
      "options": [{"option_id": String, "text": String}],
      "validation": {}
    }
  ]
}
```

设计要点：
- 使用**内嵌版本模式**（Embedded Versioning Pattern），每次修改向 `versions` 数组 `$push` 新对象，版本号通过 `$inc` 原子递增，保证并发安全。
- 版本一旦写入**不可修改、不可删除**，保证已引用该版本的问卷数据完整性。
- 恢复旧版本时基于目标旧版本内容创建新版本，不直接覆盖。
- 通过 `parent_version_number` 可回溯版本来源。
- 权限挂在最外层文档，一次查询即可获取全部版本历史和权限信息。
- 题库管理通过 `banked_by` 数组实现：加入题库 → `$addToSet`，移除 → `$pull`，查询 → `find({"access_control.banked_by": user_id})`。

索引：
- `access_control.creator`（查询"我创建的题目"）
- `access_control.shared_with`（多键索引，查询"共享给我的题目"）
- `access_control.banked_by`（多键索引，查询"我的题库"）

### 5.2 surveys.questions 从嵌入改为引用

**第一阶段（嵌入格式）：**
```json
{
  "questions": [
    {
      "question_id": "q1",
      "type": "single_choice",
      "title": "题目文本",
      "options": [...],
      "validation": {...},
      "logic": {...}
    }
  ]
}
```

**第二阶段（引用格式）：**
```json
{
  "questions": [
    {
      "question_id": "q1",
      "order": 1,
      "logic": {...},
      "question_ref_id": ObjectId,
      "version_number": 1
    }
  ]
}
```

变化说明：
- 移除了 `type`、`title`、`options`、`validation` 等题目内容字段。
- 新增 `question_ref_id`（引用 questions._id）和 `version_number`（引用的具体版本号）。
- 保留 `question_id`（问卷内部局部题号，用于跳转路由）、`order`（题目顺序）、`logic`（只属于该份问卷的跳转逻辑）。
- 新增索引：`questions.question_ref_id`（多键索引，反查"哪些问卷使用了这道题"）。

### 5.3 responses.answers 增加引用字段

**第一阶段：**
```json
{
  "answers": [
    {"question_id": "q1", "answer": "opt1"}
  ]
}
```

**第二阶段：**
```json
{
  "answers": [
    {
      "question_id": "q1",
      "question_ref_id": ObjectId,
      "version_number": 1,
      "answer": "opt1"
    }
  ]
}
```

变化说明：
- 新增 `question_ref_id` 和 `version_number`，使跨问卷统计成为可能。
- 保留原有 `question_id` 和 `answer` 字段，兼容原有逻辑。
- 新增索引：`answers.question_ref_id`（多键索引，用于跨问卷统计聚合）。

### 5.4 users 集合——未修改

users 集合结构和索引与第一阶段保持完全一致，无任何变更。

### 5.5 引用 vs 嵌入的权衡

| 维度 | 嵌入（第一阶段） | 引用（第二阶段） |
|------|------|------|
| 查询问卷+题目 | 1 次查询 | 2 次查询（先查 survey，再批量查 questions） |
| 跨问卷统计 | 需扫描所有 survey | 直接按 `answers.question_ref_id` 聚合 |
| 反查引用 | 需扫描所有 survey | 查 `surveys.questions.question_ref_id` 索引 |
| 题目共享 | 每个 survey 一份副本 | 共享同一实体，零冗余 |
| 数据一致性 | 自包含 | 依赖不变量（旧版本不删不改） |

引用方案仅多 1 次批量查询，但换来跨问卷统计、题目追踪、共享等功能的巨大便利。

### 6.5 数据迁移脚本

`backend/scripts/migrate_phase2.py` 实现一次性迁移：

1. 遍历所有问卷，从 `surveys.questions` 中抽取内嵌题目。
2. 为每个题目在 `questions` 集合中创建独立文档（版本 v1），建立 `question_id → question_ref_id` 映射。
3. 更新问卷的 `questions` 数组为引用格式（保留 `question_id`、`order`、`logic`，新增 `question_ref_id` 和 `version_number`）。
4. 更新该问卷所有答卷的 `answers`，补齐 `question_ref_id` 和 `version_number`。
5. 迁移具有幂等性：通过检查第一个题目是否已有 `question_ref_id` 判断是否已迁移，跳过已处理的问卷。

运行方式：
```bash
cd backend
python -m scripts.migrate_phase2
```

## 7. 是否遇到兼容性问题

### 7.1 一次性迁移策略

我们采用一次性迁移策略，而非编写长期兼容双结构的代码。迁移脚本将第一阶段的所有存量数据（问卷和答卷）统一转换为第二阶段结构，迁移完成后系统只需要处理引用格式，降低了代码复杂度。

### 7.2 旧测试需要适配引用格式

第一阶段的测试用例中，创建问卷时直接传入内嵌格式的题目列表。第二阶段改为引用格式后，所有测试中的题目数据都需要适配。

我们在 `backend/tests/conftest.py` 中实现了 `convert_to_refs()` 辅助函数：该函数接收旧格式的内嵌题目列表，自动在 FakeDB 中创建对应的 question 文档，并返回引用格式的题目列表。原有测试只需将题目创建方式从直接内嵌改为调用 `convert_to_refs()`，即可适配新结构，而测试的业务逻辑验证不受影响。

同时保留了 `create_base_questions_raw()` 和 `create_multi_type_questions_raw()` 函数返回旧格式数据，方便在需要原始数据的场景使用。

### 7.3 FakeDB 需要支持新操作符和 dot-notation 查询

第二阶段的服务层使用了 `$push`、`$addToSet`、`$pull`、`$inc` 等 MongoDB 更新操作符，以及 dot-notation（如 `access_control.creator`、`access_control.shared_with`）进行查询。测试框架中的 FakeDB（`conftest.py` 中的 `FakeCollection`）原本只支持 `$set` 和简单的等值查询。

为此，我们对 FakeDB 进行了扩展：
- `update_one()` 方法新增 `$inc`、`$push`、`$addToSet`、`$pull` 操作符支持。
- 新增 `_resolve_dot_path()` 和 `_set_dot_path()` 函数，支持 dot-notation 的查询和设置。
- `_matches_query()` 函数支持 dot-notation 路径匹配和 `$in` 操作符。
- 新增 `delete_one()` 方法。
- FakeDB 类新增 `questions` 集合。

## 8. 如何保证旧功能仍然可用

### 8.1 所有第一阶段测试全部通过

第一阶段共有 46 个自动化测试用例，覆盖认证（`test_auth.py`）、问卷管理（`test_surveys.py`）、答卷提交（`test_responses.py`）、统计（`test_statistics.py`）、跳转校验（`test_jump_validation.py`）五个模块。在第二阶段改造完成后，这 46 个测试用例**全部通过**，没有一个因为结构变更而失败。

这些测试的题目数据通过 `convert_to_refs()` 从旧格式转换为引用格式，但测试验证的业务逻辑（跳转路径计算、答案校验、统计聚合、状态机控制等）与第一阶段完全一致，确保了旧功能的行为不变。

### 8.3 后端解析引用返回完整内容，前端填写端无感知

这是保证旧功能透明兼容的关键设计：

- **问卷读取时**：后端自动将引用格式的题目列表解析为完整内容（批量查询 `questions` 集合获取引用版本的 `type`、`title`、`options`、`validation` 等字段），返回给前端的数据格式与第一阶段一致。
- **问卷填写时**：前端 `SurveyFill.tsx` 消费的是后端已解析的完整题目内容，不感知"引用"细节。填写体验与第一阶段完全相同。
- **答卷提交时**：后端在校验前先解析问卷引用获取题目内容，跳转逻辑计算、必填校验、答案校验全部基于解析后的内容执行，逻辑与第一阶段一致。写入时自动补齐 `question_ref_id` 和 `version_number`，对前端透明。

### 8.4 统计功能平滑过渡

- **按问卷统计**：原有的按问卷维度统计功能保持不变，后端改为先解析引用获取题目内容再聚合，返回格式不变。
- **新增跨问卷统计**：作为增量功能添加，不影响原有统计逻辑。通过 `answers.question_ref_id` 聚合所有问卷的回答数据，按 `version_number` 分组展示，同时返回命中的问卷列表。

### 8.5 问卷状态机和编辑权限不变

第一阶段的问卷状态机（draft → published → closed）和编辑权限控制完全保留：
- 草稿状态可自由编辑题目引用。
- 已发布状态冻结题目结构。
- 已关闭状态允许编辑，但题目版本改变后统计结果按版本分别展示。

## 9. 总结

第二阶段在第一阶段的三集合架构基础上，通过新增 `questions` 独立集合、将问卷题目从嵌入改为引用、答卷答案补充引用字段，实现了题目复用、共享、版本管理、题库、跨问卷统计等八项新需求。

整个改造过程严格遵循"不推倒重做"原则：
- 保留了第一阶段的核心架构（前后端分离、分层设计、统一响应体、状态机、跳转引擎、校验引擎）。
- 通过一次性数据迁移脚本处理存量数据，不写长期兼容双结构代码。
- 通过后端引用解析层屏蔽结构变更，前端填写端无感知。
- 第一阶段全部 46 个测试通过，新增 16 个测试覆盖新功能。
