# 第二阶段数据库设计文档

## 设计总览

第二阶段在第一阶段三集合（`users`、`surveys`、`responses`）基础上，主要进行了以下设计变更：

- 新增 `questions` 集合：独立存储题目，支持版本管理、共享与题库功能
- `surveys` 集合的 `questions` 数组从"嵌入完整题目内容"改为"引用题目 ID + 版本号"
- `responses` 集合的 `answers` 数组新增 `question_ref_id` 和 `version_number`，支持跨问卷统计

## 集合（Collections）设计

### 1. questions题目集合 —— 新增

独立存储题目实体，支持版本管理、共享与题库等功能。

```json
{
  // 基本信息
  "_id": ObjectId,  // id主键 题目版本谱系唯一标识
  "latest_version_number": Number,  // 最新版本号 通过 $inc 原子递增保证并发安全

  // 权限与状态
  "access_control": {
    "creator": String, // 题目最初的创建者
    "shared_with": [String],// 共享给哪些用户
    "banked_by": [String] // 哪些用户把它加入了自己的题库
  },

  // 版本与内容
  "versions": [
    {
      // 版本编号与信息
      "version_number": Number,
      "created_at": DateTime,
      "updated_by":"user_x", // 本次版本由谁保存创建
      "parent_version_number": number/null,  // 父版本号 第一版为null

      // 该版本题目内容
      "type": String,                 // 题目类型
      "title": String,                // 题目文本
      "options": [
        {"option_id": String, "text": String},
      ],
      "validation": { }               // 校验规则
    },
  ]
}
```

**索引：**

- `_id`（MongoDB 默认唯一索引，用于题目谱系定位）
- `access_control.creator`（查询"我创建的题目"）
- `access_control.shared_with`（多键索引，查询"共享给我的题目"）
- `access_control.banked_by`（多键索引，查询"我的题库"）

**设计说明：**

关于共享：

- 共享的是**同一个题目实体**，而不是复制一份新题目（保持 `_id` 不变）。
  - **为了实现跨问卷统计（需求八）：** 团队数据汇总统计，实现"这道题在全团队所有问卷中的总体数据"的聚合计算。
  - **为了题库的动态更新（统一标准）：** 如果你分享了 v1，后来更新到了 v2，同事打开"共享空间"时能看到最新版本并选择使用。
- `creator` 拥有完全控制权（创建新版本、管理共享、删除题目），`shared_with` 用户仅有**只读使用权**：
  - 可以查看题目详情及所有版本
  - 可以引用到自己的问卷
  - 可以创建**新版本**（基于原题内容）
  - **不能原地修改现有版本**（只有创建者可以）
  - 不能共享给其他人、不能取消共享、不能删除题目

  此约束的原因：原地修改会直接改变已发布问卷中的版本内容，破坏数据溯源和版本不可变原则，因此仅允许创建者为版本链增加新节点，而被共享者若想修改只能创建分支新版本。

关于版本：

- 使用**内嵌版本模式（Embedded Versioning Pattern）**，每次修改题目，不是插入新文档，而是向 `versions` 数组 `$push` 新对象，版本号通过 `$inc` 原子递增。
  - 虽然MongoDB 有单条文档 16MB 的大小限制，不过对于文本为主的问卷题目来说，即使存 1000 个版本也不会超过这个限制，所以在当前业务下是安全的，完全可以内嵌。

- 通过 `parent_version_number` 可回溯版本来源，支持 v1 → v2 → v3 的版本关系展示（可能有分支）。
- 恢复旧版本时，不直接覆盖历史版本，而是基于目标旧版本内容创建新版本。
- **版本一旦写入不可修改、不可删除**，保证已引用该版本的问卷数据完整性。
- **版本冻结保护**：若某个版本已被 `status = published` 或 `status = closed` 的问卷引用，该版本自动冻结，不允许原地修改。只有被引用版本未被任何已发布/已关闭问卷使用时，才允许原地修改；否则必须创建新版本。这确保已发布和已关闭问卷的版本数据永远不变。
- 权限挂在最外层，**权限控制简单**
- 找历史记录不需要额外查表，一次查询全部拿到

关于并发安全：

```python
# 创建新版本时使用原子操作，保证 version_number 不冲突
result = db.questions.find_one_and_update(
  {"_id": ObjectId("680000000000000000000001")},
    {
        "$inc": {"latest_version_number": 1},
        "$push": {"versions": new_version_obj}
    },
    return_document=ReturnDocument.AFTER
)
```

关于题库（不需要独立 collection）：

- "题库管理"本质是用户的个人收藏夹：加入题库 → `$addToSet banked_by`，移除 → `$pull banked_by`，查询 → `find({"access_control.banked_by": user_id})`。
- 当前需求不涉及"创建多个命名分类题库"，`banked_by` 方案完全满足，不需要独立collection。

关于题目来源分类：

用户视角下的题目按来源分为三类：

1. **我的题目**（`access_control.creator == user_id`）：当前用户创建的所有题目，无论是否加入题库。拥有完全控制权（编辑、版本管理、共享、删除）。
2. **共享给我的题目**（`access_control.shared_with` 包含 `user_id` 且 `creator != user_id`）：他人共享的题目，只读使用权（查看、引用、创建新版本、加入题库）。
3. **我的题库**（`access_control.banked_by` 包含 `user_id`）：用户主动加入的题目集合（包含自创和被共享的题目），作为个人的题目收藏夹。可从题库移除，但题目本身不被删除。

### 2. surveys问卷集合 —— questions 部分修改为引用

问卷的 `questions` 数组从**嵌入完整题目**内容改为引用 `question_ref_id + version_number`来确定题目内容，仅保留题目针对问卷的局部上下文。

```json
{
  "_id": ObjectId,
  "title": String,
  "description": String,
  "creator_id": ObjectId,
  "access_code": String,
  "status": String,  // "draft" / "published" / "closed"
  "created_at": DateTime,
  "updated_at": DateTime,
  "deadline": DateTime,
  "response_count": Number,
  "settings": {
    "allow_anonymous": Boolean,
    "allow_multiple": Boolean
  },

  "questions": [
    {
      // 局部上下文（仅本问卷有效，不污染全局题库）
      "question_id": String,     // 问卷内部局部题号（如 "q1"），给 logic 跳转路由用
      "order": Number,           // 题目在此问卷中的顺序
      "logic": {                 // 跳转逻辑（只属于该份问卷）
        "enabled": Boolean,
        "rules": [...]
      },

      // 题目溯源（用 question_ref_id + version_number 唯一确定题目内容）
      "question_ref_id": ObjectId, // 引用 questions._id
      "version_number": Number   // 引用的具体版本号
    }
  ]
}
```

**索引：**

- 新增`questions.question_ref_id`（多键索引，反查"哪些问卷使用了这道题"——需求六）
- 其余索引与第一阶段保持不变

**设计说明：**

关于引用 vs 嵌入的选择：

| 维度                 | 嵌入（第一阶段）     | 引用 question_ref_id + version_number（第二阶段）  |
| -------------------- | -------------------- | -------------------------------------------------- |
| 查询问卷+题目        | 1 次查询             | 2 次查询（先查 survey，再 `$in` 批量查 questions） |
| 跨问卷统计（需求八） | 需扫描所有 survey    | 直接按 `responses.answers.question_ref_id` 聚合    |
| 反查引用（需求六）   | 需扫描所有 survey    | 查 `surveys.questions.question_ref_id` 索引        |
| 题目共享（需求二）   | 每个 survey 一份副本 | 共享同一实体，零冗余                               |
| 数据一致性           | 自包含               | 依赖不变量（旧版本不删不改）                       |

引用方案仅多 1 次批量查询，但换来跨问卷统计、题目追踪、共享等功能的巨大便利，因此设计修改为引用问题内容

关于题目修改与删除：

- 每次保存的新修改都会自动作为题目的新 version 保存；已使用旧版本的其他问卷保持原引用版本不变。
- **已发布状态（`published`）的问卷题目结构冻结**：无法编辑题目结构（增删题目、调整顺序）和元信息，确保正在收集中的问卷稳定。
- **草稿（`draft`）和关闭（`closed`）状态的问卷可编辑**：可修改题目结构、元信息，可升级引用的题目版本号。
- **题目删除保护**：删除题目前必须检查是否有已发布或已关闭的问卷正在使用该题目的任何版本。若存在，删除被拒绝（返回错误）。仅当题目无被已发布/已关闭问卷引用时，才允许删除，同时从所有草稿问卷中级联移除该题目引用。
- 如果发布后又关闭，旧版本题目已收集到回复后进行了修改，统计结果需按版本分别展示，不能统一覆盖（配合 `responses` 中的版本绑定实现）。

关于`question_id`、`order`和`question_ref_id`的说明：

- `question_id`是问卷内部局部题号（如 "q1"），给 logic 跳转路由用
- `order`表示问卷的题目标号，题目顺序
- `question_ref_id`引用 `questions._id`，是问卷对题目的外键，题目对象的唯一标识号
- 无法用`question_ref_id`代替`question_id`：用户如果把同一个题库模板添加两次，分别用来评价“服务态度”和“专业能力”。只使用全局的 `question_ref_id` 会导致两道题在问卷拥有完全相同的 ID，跳转错误
- 无法用`order`代替`question_id`：假设跳转逻辑是：“如果选 A，跳转到第 8 题”。 如果用户在编辑问卷时，把第 5 题**拖拽**到了第 9 题的位置，那么原来第 8 题的 `order` 就变成了 7。如果用 `order` 跳转，就会跳到现在的第 8 题（原来的第 9 题），逻辑错位。导致每次调整题目顺序，都必须把所有牵涉到的跳转逻辑改一遍，不合适。

### 3. responses答卷集合 —— 新增回答与题目版本的绑定

答卷的 `answers` 数组新增 `question_ref_id` 和 `version_number`，支持跨问卷统计和版本追踪。

```json
{
  "_id": ObjectId,
  "survey_id": ObjectId,
  "respondent_id": ObjectId, // 匿名时为 null
  "is_anonymous": Boolean,
  "submitted_at": DateTime,
  "answers": [
    {
      "question_id": String,     // 问卷内部局部题号
      "question_ref_id": ObjectId,  // 新增：引用 questions._id，方便跨问卷统计
      "version_number": Number, // 新增：记录回答对应的题目版本号
      "answer": Mixed  // 答案值（与第一阶段格式一致）
    }
  ],
  "completion_time": Number
}
```

**索引：**

- 新增`answers.question_ref_id`（多键索引，跨问卷统计——需求八）
- 其余索引与第一阶段保持不变

**设计说明：**

- `question_ref_id` + `version_number` 使跨问卷统计成为可能：按 `question_ref_id` 聚合所有问卷的回答，按 `version_number` 分组可观察不同版本的数据差异。

  "password_hash": String,   // 密码哈希值

**索引：** `username`（唯一索引）
