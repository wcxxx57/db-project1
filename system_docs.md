# 问卷系统 - 系统说明

## 1. 项目概述

本项目是一个基于 **MongoDB** 的在线问卷系统，目标是实现一个简化版的问卷平台，支持用户注册登录、创建问卷、发布问卷、填写问卷以及查看统计结果等核心功能。

系统重点围绕以下几个方面展开：
- 使用 MongoDB 完成问卷系统的数据建模
- 使用后端程序实现问卷创建、答卷提交、统计分析等业务逻辑
- 支持题目校验规则与跳转逻辑
- 使用 AI 工具辅助系统设计、代码生成与文档整理

本系统为课程项目第一阶段成果，后续还需要在此基础上进行第二阶段的需求变更与扩展，因此在设计时会尽量保证结构清晰、易于修改和扩展。

## 3. 技术选型说明

### 3.1 后端：Python + FastAPI

后端采用 Python + FastAPI 进行开发。FastAPI 具有以下优点：
- 开发效率高，语法简洁，适合课程项目快速实现
- 原生支持类型标注，便于定义清晰的数据结构
- 自动生成 Swagger/OpenAPI 文档，方便调试与接口联调
- 适合实现问卷系统中的注册登录、问卷管理、答卷提交与统计接口

### 3.2 数据库：MongoDB + PyMongo

数据库采用 MongoDB，驱动选择 PyMongo。原因如下：
- 问卷、题目、校验规则、跳转逻辑都具有灵活嵌套结构，适合文档型数据库建模
- 题目嵌入问卷、答案嵌入答卷记录，能减少关联查询，提高读取效率
- MongoDB 的 schema 灵活，便于第二阶段需求变更时扩展字段
- PyMongo 使用简单直接，适合当前项目开发节奏

### 3.3 前端：React + Vite + TypeScript

前端采用 React + Vite + TypeScript。原因如下：
- React 适合实现动态表单、题目编辑器、问卷填写流程等交互较多的页面
- Vite 启动快、构建轻量，适合课程项目快速开发
- TypeScript 能提升代码可维护性，减少前后端字段对接错误

### 3.4 UI：shadcn/ui + Tailwind CSS

前端界面使用 shadcn/ui 与 Tailwind CSS 搭建。原因如下：
- shadcn/ui 组件风格简洁，适合快速搭建不复杂但规范的页面
- Tailwind CSS 可高效完成布局与样式调整
- 结合 React 使用方便，适合登录、问卷编辑、填写、统计等页面开发

### 3.5 认证：JWT（PyJWT）

系统认证采用 JWT。原因如下：
- 登录后由后端签发 token，前端后续请求携带 token 即可完成身份验证
- 适合前后端分离架构
- 便于区分需要登录访问的接口与公开访问的问卷填写接口

## 4. 系统功能概述

### 4.1 用户账号功能
- 用户注册（用户名、密码、邮箱）
- 用户登录（JWT 认证）
- 登录后查看自己创建的问卷

### 4.2 问卷创建与管理
- 创建问卷（标题、说明、是否允许匿名填写）
- 编辑问卷中的题目（单选题、多选题、文本填空、数字填空）
- 为题目设置校验规则（选择数量限制、字数限制、数值范围限制等）
- 为题目设置跳转逻辑（根据单选/多选/数字填空结果跳转到指定题目）
- 发布问卷 / 关闭问卷
- 设置问卷截止时间
- 通过访问码生成分享链接（`/survey/{access_code}`）

### 4.3 填写问卷
- 通过分享链接进入问卷
- 按顺序回答题目，支持跳转逻辑
- 实时校验填写内容是否符合规则
- 所有填写均需先登录，支持设置问卷的匿名结果显示模式
- 支持多人同时填写
- 支持同一用户多次提交（若问卷允许）

### 4.4 统计结果查看
- 单选题：每个选项的选择人数
- 多选题：每个选项的选择次数
- 文本填空：查看所有填写内容
- 数字填空：查看所有填写内容 + 平均值
- 查看整个问卷的统计 / 查看某一道题的统计

## 7. 问卷修改策略

### 7.1 设计决策

本系统允许在 `closed`（已关闭）状态下修改问卷，包括题目结构，以提供最大的灵活性。

**状态与编辑权限：**
- `draft` 状态：可以自由编辑所有内容（未发布，无答卷数据）
- `published` 状态：不可编辑（正在收集答卷，保证数据一致性）
- `closed` 状态：可以编辑（已停止收集，创建者可能需要修正错误或调整问卷）

### 7.2 潜在风险与应对

**风险：**
修改题目结构后，历史答卷中的 `question_id` 或 `option_id` 可能找不到对应定义，导致统计结果不准确。

**具体场景：**
1. 创建者删除了某个题目（如 q2），但历史答卷中仍有 q2 的答案
2. 创建者修改了选项 ID 或删除了某个选项，但历史答卷中仍引用旧的 option_id
3. 创建者修改了题目类型，导致答案格式不匹配

**应对机制：**

1. **统计代码容错逻辑**（已实现）：
   - 统计时自动检测孤儿答案（对应题目已被删除的答案）
   - 统计时自动检测未知选项（对应选项已被删除的答案）
   - 在统计结果中添加 `warning` 字段，告知用户数据不完整的原因
   - 示例：`"warning": "有 5 个答案的选项在当前题目定义中已不存在（可能是历史数据）"`

2. **前端警告提示**（需实现）：
   - 在编辑页面顶部显示警告横幅
   - 当 `response_count > 0` 时提示用户谨慎修改题目结构
   - 建议用户只修改标题、说明等元数据，避免修改题目结构

3. **API 文档说明**（已完成）：
   - 在 `PUT /surveys/{survey_id}` 接口文档中明确说明修改风险
   - 提供最佳实践建议

### 7.3 最佳实践建议

**对于创建者：**
- 如果问卷已有答卷数据（`response_count > 0`），建议：
  - 只修改标题、说明、截止时间等元数据
  - 避免删除题目或修改选项 ID
  - 如需大幅修改题目结构，建议创建新问卷

**对于系统使用者：**
- 在发布问卷前仔细检查题目设置
- 发布后如发现错误，先关闭问卷，谨慎修改
- 查看统计结果时注意警告信息

### 7.4 为第二阶段预留的扩展点

如果第二阶段需求要求更严格的数据一致性保护或更灵活的修改能力，可以考虑：

1. **版本管理机制**：
   - 每次修改题目结构时创建新版本
   - 答卷记录关联到具体版本（`question_version` 字段）
   - 统计时按版本分别统计，或合并统计时智能匹配题目

2. **问卷分支（fork）功能**：
   - 修改时自动创建新问卷
   - 保留原问卷及其答卷数据不变

3. **严格保护模式**：
   - 增加配置选项，当 `response_count > 0` 时完全禁止修改题目结构
   - 只允许修改元数据

4. **答卷快照**：
   - 在答卷记录中保存题目定义的快照
   - 统计时使用快照数据，不受后续修改影响

## 8. 开发方式说明

本项目采用**前后端分离**方式开发：
- 后端负责数据存储、业务逻辑、接口实现、校验和统计处理
- 前端负责页面展示、交互逻辑、表单收集与接口调用
- 前后端通过 RESTful API 通信，使用 JSON 格式传输数据
- 认证通过 JWT token 在 HTTP 请求头中传递
