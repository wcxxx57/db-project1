# 关键逻辑说明

本文档主要说明系统中最为核心的关键业务机制：从动态的题目跳转与校验到匿名权限解耦、并发防刷与容错设计。这些机制体现了借助 MongoDB schema-less 特性在复杂业务场景下的思考深度。

---

## 1. 题目跳转逻辑

为了实现类似"问卷星"的动态答题体验，系统在后端实现了一套基于条件分支的跳转引擎。主要分为"问卷定义期的合法性校验"与"用户答卷时的路径计算"两部分。

### 1.1 逻辑定义结构
题目模型（`QuestionLogic`）支持为每个题目单独配置跳转规则，每条规则由"触发条件"和"执行动作"组成：

**触发条件 (Condition) 的三种类型：**
- **`select_option`（单选匹配）：** 当用户选择了指定的某个选项时触发。例如："您的年龄段？" 选择 "18岁以下" 时跳过后续成人内容题目。
- **`contains_option`（多选包含）：** 当用户的多选答案中包含指定选项时触发，支持 `any`（包含任一）和 `all`（包含全部）两种匹配模式。例如："您感兴趣的领域？" 选择了 "编程" 时跳转到技术相关题目。
- **`number_compare`（数字比较）：** 支持 7 种比较操作符（`eq`/`ne`/`gt`/`gte`/`lt`/`lte`/`between`），用于数值题的条件判断。例如："您的月收入？" 大于 10000 时跳转到高端消费调查。

**执行动作 (Action) 的两种类型：**
- **`jump_to`：** 跳转到指定的 `target_question_id`，跳过中间所有题目。
- **`end_survey`：** 强制结束问卷，用户无需回答后续任何题目（例如筛选不符合条件的受访者）。

### 1.2 多重规则冲突解决机制 
在多选题或同题多条条件判断的场景中，用户提交的答案可能会同时满足多条不同的跳转规则。

**冲突场景示例：**
假设题目"您感兴趣的领域？"有三条跳转规则：
- 规则 1：选择了"编程" → 跳转到题目 Q5（技术调查）
- 规则 2：选择了"设计" → 跳转到题目 Q8（创意调查）
- 规则 3：选择了"管理" → 结束问卷

当用户同时选择了"编程"和"设计"时，三条规则中的前两条都满足，此时应该跳转到 Q5 还是 Q8？

**解决策略：首条规则匹配优先 (First-Match Priority)**

`response_service.py` 中的 `compute_jump_target()` 函数采用顺序短路匹配策略：
```python
for rule in rules:
    condition = rule.get("condition", {})
    action = rule.get("action", {})
    if evaluate_condition(condition, answer):
        # 第一个命中的规则立即生效，后续规则不再评估
        if action_type == "end_survey":
            return "__END__"
        elif action_type == "jump_to":
            return action.get("target_question_id")
```

**设计优势：**
- **行为确定性：** 严格按照问卷结构中定义的规则数组顺序从上至下判定，保证相同输入产生相同输出。
- **短路评估：** 一旦用户答案命中了某一条规则，引擎将立刻执行该动作，并**直接中断**对该题后续规则的评估，避免不必要的计算。
- **设计者控制权：** 将核心控制权交给问卷设计者，遇互斥重叠条件时，仅需将"高优先级"规则前置即可，兼顾轻量与逻辑严密性。

### 1.3 问卷保存期的安全性校验
鉴于跳转容易引起死胡同或死循环，系统在保存环节即执行严密校验。`survey_service.py` 中的 `_validate_jump_logic()` 函数实现了三层防护：

**第一层：目标存在性校验**
```python
qid_set = {q["question_id"] for q in questions}
if target and target not in qid_set:
    return f"「{q.get('title')}」的跳转目标「{target}」不存在"
```
- 构建题目 ID 集合，确保 `jump_to` 目标 ID 必须真实存在于当前题目列表。
- 防止因手动输入错误或删除题目后未清理跳转规则导致的"跳转到虚空"问题。

**第二层：禁止向前跳转**
```python
qid_to_order = {q["question_id"]: q.get("order", 0) for q in questions}
if target and qid_to_order.get(target, 0) <= qid_to_order.get(q["question_id"], 0):
    return f"「{q.get('title')}」不允许向前跳转到「{target}」"
```
- 构建题目 ID 到序号（`order`）的映射表，确保目标题目的序号必须严格大于当前题目。
- 从根本上阻断向前跳转，避免用户在答题时陷入"回到之前题目"的混乱状态。

**第三层：环路检测 (Cycle Detection)**
```python
def has_cycle(start_qid: str, visited: set) -> bool:
    if start_qid in visited:
        return True  # 检测到环路
    visited.add(start_qid)
    # 递归检查所有可能的跳转目标
    for rule in logic.get("rules", []):
        if action.get("type") == "jump_to":
            target = action.get("target_question_id")
            if target and has_cycle(target, visited.copy()):
                return True
    return False
```
- 系统运用深度优先搜索（DFS）算法开展有向图环路检测。
- 即使是间接循环（Q1 → Q3 → Q5 → Q3），也会被识别并拒绝保存。
- 错误提示会明确指出涉及循环的题目，方便问卷设计者定位问题。

### 1.4 回答阶段的路径推导引擎
在用户正式提交答卷时，后端的防伪路径推导开始执行。`response_service.py` 中的 `compute_required_questions()` 函数实现了完整的路径计算：

**路径计算流程：**
```python
def compute_required_questions(questions: List[Dict], answers_map: Dict[str, Any]) -> List[str]:
    required = []
    idx = 0
    while idx < len(questions):
        q = questions[idx]
        qid = q["question_id"]
        required.append(qid)
        
        # 获取用户对该题的答案
        answer = answers_map.get(qid)
        if answer is not None:
            jump_target = compute_jump_target(q, answer)
            if jump_target == "__END__":
                break  # 结束问卷
            elif jump_target:
                # 跳转到目标题目
                target_idx = next((i for i, q in enumerate(questions) if q["question_id"] == jump_target), None)
                if target_idx is not None:
                    idx = target_idx
                    continue
        idx += 1
    return required
```

**核心机制：**
- **动态路径计算：** 从第一题开始，根据用户的实际答案逐题推导，生成 `required_qids` 列表（本次答卷实际应当回答的题目 ID 列表）。
- **跳转执行：** 遇到 `jump_to` 时直接跳转到目标题目索引，跳过中间所有题目；遇到 `end_survey` 时立即终止循环。
- **脏数据过滤：** 提交校验时，只有 `required_qids` 列表内的题目才被承认有效并要求"必答校验"。若用户通过截包或篡改等异常手段提交了被跳过题目的野答案，引擎将直接静默抛弃之，保证逻辑自洽。

**防篡改示例：**
假设问卷有 Q1、Q2、Q3、Q4 四题，Q1 选择"A"时跳转到 Q4。
- 正常用户：提交 `{Q1: "A", Q4: "答案"}`，路径推导得到 `required_qids = [Q1, Q4]`，校验通过。
- 恶意用户：提交 `{Q1: "A", Q2: "伪造", Q3: "伪造", Q4: "答案"}`，路径推导仍得到 `required_qids = [Q1, Q4]`，Q2 和 Q3 的答案被识别为"不在必答列表中"而被静默丢弃。

## 2. 动态数据校验逻辑 

结合 Pydantic 模型，系统开发了高度参数化且基于数据驱动的表单校验引擎。不同题型的校验规则完全由数据库中的 `validation` 字段驱动，无需硬编码。

### 2.1 校验规则模型
`QuestionValidation` 对象可按不同题型携带不同制约：

**多选题（`multiple_choice`）：**
```python
validation = {
    "min_selected": 2,  # 至少选择 2 项
    "max_selected": 5,  # 最多选择 5 项
    "exact_selected": None  # 或精确要求选择 N 项
}
```
- 支持"至少选几项"、"最多选几项"、"精确选几项"三种约束。
- 例如："请选择 2-3 个您最喜欢的功能"。

**文本题（`text_input`）：**
```python
validation = {
    "min_length": 10,   # 最少输入 10 个字符
    "max_length": 500   # 最多输入 500 个字符
}
```
- 防止用户敷衍（最小字数）和灌水（最大字数）。
- 例如："请详细描述您的建议（至少 50 字）"。

**数字题（`number_input`）：**
```python
validation = {
    "min_value": 0,       # 最小值
    "max_value": 150,     # 最大值
    "integer_only": True  # 是否只允许整数
}
```
- 限制数值范围，防止异常输入（如年龄输入 -10 或 999）。
- 支持整数约束（如"请输入您的年龄"只能是整数）。

### 2.2 校验管线与防篡改
验证包含两个阶段，分别在问卷保存期和答卷提交期执行：

**阶段一：配置自查期（问卷保存时）**

`survey_service.py` 中的 `_validate_question_validation()` 函数在保存问卷时检查配置合理性：
```python
# 多选题验证
if q_type == "multiple_choice":
    min_sel = validation.get("min_selected")
    max_sel = validation.get("max_selected")
    if min_sel is not None and max_sel is not None and max_sel < min_sel:
        return f"「{q_title}」最多选择项({max_sel})不能小于最少选择项({min_sel})"
```
- 确认最值关系必须合理（`max_length` 不能小于 `min_length`、`max_value` 不能小于 `min_value` 等）。
- 在问卷设计阶段就拦截配置错误，避免用户答题时遇到无法满足的矛盾约束。

**阶段二：提交运行时（答卷提交时）**

`response_service.py` 中的 `_validate_answer()` 函数在用户提交时执行实际校验：
```python
# 选择题防篡改：剔除不在备选池中的伪造 option_id
valid_option_ids = {opt["option_id"] for opt in question.get("options", [])}
if q_type == "single_choice":
    if answer not in valid_option_ids:
        raise ResponseServiceError(ErrorCodes.INVALID_ANSWER, 
            f"「{q_title}」的选项不合法", 400)
elif q_type == "multiple_choice":
    if not isinstance(answer, list):
        raise ResponseServiceError(...)
    # 过滤掉伪造的 option_id
    answer = [opt for opt in answer if opt in valid_option_ids]
```
- **选择题防篡改：** 引擎接管用户的完整 Payload，在提取 `option_id` 时与题目定义的 `options` 列表比对，剔除不在备选源池中的伪造 ID。
- **数值范围校验：** 对文本长度、数字大小、选择项数等进行判断。
- **错误定位：** 如遇任何超限或异常，立刻抛出 `400 Bad Request` 和精确的错误信息（包含题目标题），方便用户定位问题。

## 3. 匿名提交与权限解耦逻辑 

为了兼顾"防止恶意刷单"与"保护填写者隐私"，系统实现了鉴权控制与展示数据的解耦。这是一个三层防护架构。

### 3.1 第一层：JWT 鉴权始终生效
`routes/responses.py` 中的提交接口使用 `Depends(get_current_user)` 装饰器：
```python
@router.post("/surveys/{survey_id}/responses")
async def submit_response(
    survey_id: str,
    current_user: dict = Depends(get_current_user),  # JWT 必须验证
    ...
):
```
- **无论问卷是否允许匿名**，后端在提交流程的最前置位均要求验证 JWT 登录状态。
- 这确保了每份答卷都能追溯到真实用户（用于防刷、审计），同时为后续的匿名脱敏提供了基础。

### 3.2 第二层：物理级别的匿名脱敏
`response_service.py` 中的 `submit_response()` 函数根据问卷设置和用户意愿决定是否匿名：
```python
# 判断是否匿名提交
is_anonymous = survey.get("allow_anonymous", False) and request_anonymous
response_doc = {
    "survey_id": survey_id,
    "respondent_id": current_user["_id"],  # 物理层始终记录真实 ID
    "is_anonymous": is_anonymous,          # 标记是否匿名
    "answers": validated_answers,
    ...
}
```
- 若问卷设定（`allow_anonymous=True`）满足且提交者宣称匿名（`request_anonymous=True`），记录落地的 `is_anonymous` 被标记为 `True`。
- **关键设计：** `respondent_id` 在物理层始终记录真实用户 ID（用于防刷和审计），但通过 `is_anonymous` 标记控制是否在统计层展示。

### 3.3 第三层：统计层的强制阻断
`statistics_service.py` 中的 `get_survey_statistics()` 在构建统计结果时执行严格的隐私屏蔽：
```python
for doc in all_responses:
    respondent_id = doc.get("respondent_id")
    is_anon = doc.get("is_anonymous", True)
    
    respondent_info = {
        "respondent_id": None if is_anon else str(respondent_id),
        "display_name": "匿名用户" if is_anon else user_map.get(str(respondent_id), "未知用户"),
        "is_anonymous": is_anon
    }
```
- 将所有 `is_anonymous=True` 的受访者 ID 强制置为 `None`。
- 将 `display_name` 抹去为"匿名用户"。
- 彻底切断了利用前端 API 挖掘人员私密信息的可能性，即使攻击者拦截网络请求也无法获取匿名用户的真实身份。

**三层架构的优势：**
- **防刷能力：** 物理层记录真实 ID，可以检测同一用户的重复提交（`allow_multiple=false` 时）。
- **隐私保护：** 统计层完全屏蔽匿名用户身份，前端无法获取。
- **审计追溯：** 管理员在数据库层面仍可追溯匿名答卷的真实提交者（用于处理恶意提交）。

## 4. "读多写少"场景的性能冗余优化

我们采用了反范式（Denormalization）冗余策略应对典型的"读多写少"场景。

### 4.1 问题识别：Dashboard 列表的性能瓶颈
**痛点场景：**
用户打开首页 Dashboard，需要展示所有问卷的列表，每个问卷卡片显示：
- 问卷标题、描述
- 创建时间
- **已收集答卷数**（关键指标）

如果采用规范化设计，获取答卷数需要：
```python
# 低效做法：对每个问卷都执行一次 count 查询
for survey in surveys:
    count = db.responses.count_documents({"survey_id": survey["_id"]})
    survey["response_count"] = count
```
- 假设用户有 50 个问卷，需要执行 51 次数据库查询（1 次查问卷列表 + 50 次 count）。
- 每次 `count_documents` 需要扫描 `responses` 集合（即使有索引，也需要遍历索引）。
- 在答卷数量达到百万级时，count 操作会变得非常慢。

### 4.2 解决方案：冗余字段 + 原子更新
在 `surveys` 文档内直接存储 `response_count` 字段：
```python
survey_doc = {
    "_id": ObjectId(...),
    "title": "用户满意度调查",
    "response_count": 0,  # 冗余字段，实时维护
    ...
}
```

**关键实现：** `response_service.py` 中提交答卷时的原子更新：
```python
# 插入答卷
result = db.responses.insert_one(response_doc)

# 原子递增问卷的答卷计数
db.surveys.update_one(
    {"_id": ObjectId(survey_id)},
    {"$inc": {"response_count": 1}}  # 原子操作，线程安全
)
```

### 4.3 原子操作的并发安全性
MongoDB 的 `$inc` 操作符是原子的，即使多个用户同时提交答卷，也不会出现计数错误：
- **场景：** 用户 A 和用户 B 同时提交答卷，当前 `response_count = 100`。
- **传统做法（非原子）：** 
  1. A 读取 `count = 100`，计算 `count + 1 = 101`，写回 101。
  2. B 读取 `count = 100`（A 还未写回），计算 `count + 1 = 101`，写回 101。
  3. 结果：两次提交后 `count = 101`（错误，应该是 102）。
- **原子操作：** MongoDB 在数据库层面保证 `$inc` 的原子性，两次提交后 `count = 102`（正确）。

### 4.4 性能收益量化
- **查询次数：** Dashboard 列表从 O(n) 次查询降低到 1 次查询（n 为问卷数）。
- **响应时间：** 从约 5 秒（50 个问卷 × 100ms count）降低到约 50ms（单次查询问卷列表）。
- **写入开销：** 每次提交答卷增加一次 `$inc` 操作（约 1ms），几乎可以忽略。

**典型的"读多写少"权衡：** 用极小的写入开销（每次提交多 1ms），换取巨大的读取性能提升（列表查询快 100 倍）。

## 5. 结构修改容错与数据安全边界 

系统开放了 `closed`（已停止收集）状态下的题型重排与二次编辑自由度。但这极易出现因旧的历史答卷挂空产生的"孤儿数据"崩溃。为此配置了强壮的容错设计。

### 5.1 问卷状态机与编辑权限控制
问卷有三种状态，每种状态下的编辑权限不同：

**状态转换规则：**
```
draft (草稿) → published (已发布) → closed (已停止收集)
     ↓              ↓                    ↓
  可编辑         禁止编辑            可编辑（有限）
```

**`survey_service.py` 中的状态锁实现：**
```python
def update_survey(survey_id: str, update_data: dict, current_user: dict):
    survey = db.surveys.find_one({"_id": ObjectId(survey_id)})
    
    # 状态锁：published 状态禁止修改结构
    if survey.get("status") == "published":
        # 只允许修改状态字段（如切换到 closed）
        if any(key in update_data for key in ["title", "description", "questions"]):
            raise SurveyServiceError(
                ErrorCodes.INVALID_STATUS,
                "已发布的问卷不允许修改结构，请先停止收集",
                400
            )
```

**设计原因：**
- **`draft` 状态：** 问卷尚未发布，没有答卷数据，可以随意修改。
- **`published` 状态：** 问卷正在收集答卷，此时修改结构会导致：
  - 已提交的答卷与新结构不匹配（孤儿数据）。
  - 正在填写的用户看到的题目突然变化（用户体验极差）。
  - 跳转逻辑可能失效（删除了跳转目标题目）。
- **`closed` 状态：** 问卷已停止收集，不会再有新答卷，可以安全修改结构（虽然会产生孤儿数据，但统计引擎有容错机制）。

### 5.2 容错设计：统计时的孤儿数据处理
当问卷在 `closed` 状态下被修改后，统计引擎需要处理历史答卷中的孤儿数据。详见第 9 节的完整实现。

**核心策略：**
- **聚合拦截跳过：** 生成统计分析时，引擎将逐个 `answer` 交叉校验当下的最新 schema，若对应题目已被抹除，统计系统则静默跳过该废项，而不产生整体分析中止 500 级错误。
- **警告提示：** 在统计结果中添加 `warning` 字段，告知问卷创建者存在孤儿数据。

### 5.3 为什么不采用数据迁移方案
**备选方案：** 当问卷结构修改时，自动更新所有历史答卷的数据结构。

**拒绝理由：**
- **复杂度高：** 需要编写复杂的数据迁移逻辑，处理各种边界情况（删除题目、删除选项、修改题型等）。
- **风险大：** 自动修改历史数据可能导致数据丢失或错误，且难以回滚。
- **性能差：** 如果问卷有数万份答卷，批量更新会非常慢，且会锁表影响其他操作。
- **不必要：** 问卷在 `closed` 状态下修改结构是低频操作，且统计引擎已有容错机制，无需强制迁移。

**当前方案的优势：** 保持历史数据不变（审计友好），在统计时动态适配（灵活性高），通过警告提示告知问题（透明度高）。

## 6. 前端跳转逻辑引擎与实时预览机制

为保证用户在填写问卷时的所见即所得体验，前端实现了与后端完全一致的跳转逻辑引擎。

### 6.1 前后端跳转引擎的镜像设计
前端 `SurveyFill.tsx` 中的三个核心函数与后端 `response_service.py` 保持逻辑同构：
- **`evaluateCondition(condition, answer)`：** 评估单个跳转条件是否成立，支持三种条件类型（单选匹配、多选包含、数字比较）。
- **`computeJumpTarget(question, answer)`：** 根据题目的跳转规则和用户答案，按规则顺序评估，返回跳转目标（`"__END__"` / `question_id` / `null`）。
- **`computeVisibleQuestions(questions, answersMap)`：** 从第一题开始，根据每题跳转逻辑和当前所有答案，动态计算用户实际需要回答的题目序列。

### 6.2 长列表滚动模式的交互优势
在"单题翻页"与"长列表滚动"两种交互模式的权衡中，最终选择了长列表模式：
- **跳转可视化：** 当用户修改某题答案触发跳转时，后续题目会实时出现或消失，用户可以直观感知到跳转逻辑的生效。
- **状态管理简化：** 避免了单题翻页模式下因跳转导致的"当前页码失效"、"需要回退到哪一页"等复杂状态维护问题。
- **组件生命周期清晰：** 每道题目的渲染完全由 `visibleQuestions` 数组驱动，React 自动处理 DOM 的增删，无需手动管理题目组件的挂载/卸载。

### 6.3 实时预览的性能优化
- **答案变更时的增量计算：** 每次用户修改答案，`useMemo` 钩子会重新计算可见题目列表，但只有当 `answersMap` 真正变化时才触发重算。
- **防止向前跳转的前端保护：** 虽然后端已有向前跳转检测，但前端引擎在计算路径时也会主动跳过向前跳转指令（`if (qidToIndex[target] > idx)`），双重保障避免死循环。

## 7. 索引设计与查询优化策略

MongoDB 的索引设计直接影响查询性能。系统在 `database.py` 的 `init_indexes()` 中定义了五个核心索引，覆盖所有高频查询场景。

### 7.1 users 集合索引
```python
db.users.create_index("username", unique=True)
```
- **用途：** 注册时的用户名唯一性校验、登录时的用户名查找。
- **唯一索引保证：** 数据库层面阻止重复用户名，避免并发注册导致的竞态条件（两个请求同时检查用户名不存在，然后都尝试插入）。

### 7.2 surveys 集合索引
```python
db.surveys.create_index("creator_id")
db.surveys.create_index("access_code", unique=True)
db.surveys.create_index("status")
db.surveys.create_index([("created_at", -1)])
```
- **`creator_id` 索引：** 支持"我的问卷列表"查询（`GET /surveys/my`），按创建者过滤。高频操作，必须建索引。
- **`access_code` 唯一索引：** 支持公开访问入口（`GET /public/surveys/{access_code}`）的快速定位，同时保证访问码全局唯一，避免碰撞。
- **`status` 索引：** 支持按状态筛选问卷（如仅查询已发布问卷），为第二阶段可能的"问卷市场"功能预留。
- **`created_at` 降序索引：** 支持问卷列表按创建时间倒序排列（最新问卷在前），提升用户体验。

### 7.3 responses 集合索引
```python
db.responses.create_index([("survey_id", 1), ("submitted_at", -1)])
db.responses.create_index([("survey_id", 1), ("respondent_id", 1)])
db.responses.create_index("respondent_id")
```
- **`survey_id + submitted_at` 复合索引：** 支持统计聚合时按问卷 ID 查询所有答卷，并按提交时间排序（最新答卷在前）。统计是最高频操作，此索引至关重要。
- **`survey_id + respondent_id` 复合索引：** 支持重复提交检查（`allow_multiple=false` 时查询该用户是否已提交过该问卷）。每次提交都需要执行此查询，必须建索引。
- **`respondent_id` 单独索引：** 支持查询某用户的所有答卷历史（为第二阶段可能的"我的答卷"功能预留）。

### 7.4 复合索引的字段顺序选择
MongoDB 复合索引遵循"最左前缀"原则。以 `(survey_id, submitted_at)` 为例：
- 查询 `survey_id = X` 时可使用该索引（只用到第一个字段）。
- 查询 `survey_id = X AND submitted_at > Y` 时可完全使用该索引（用到两个字段）。
- 但查询 `submitted_at > Y`（不带 `survey_id`）时无法使用该索引。

**字段顺序的选择依据：** 将过滤性最强、查询频率最高的字段放在前面。在统计场景中，总是先按 `survey_id` 过滤（将百万级答卷缩小到某问卷的几百份），再按时间排序，所以 `survey_id` 在前。

### 7.5 为什么不建更多索引
索引并非越多越好：
- **写入性能代价：** 每个索引都会增加插入/更新操作的开销（需要同步更新索引）。
- **存储空间占用：** 索引本身需要占用磁盘空间。
- **维护成本：** 过多索引会增加数据库维护的复杂度。

因此只为高频查询场景建立索引，低频查询（如按 IP 地址查询答卷）可以接受全表扫描。

## 8. 统计聚合中的批量查询优化

统计模块需要展示答题者的用户名，如果对每个答卷都单独查询一次 `users` 集合，会产生经典的 N+1 查询问题。

### 8.1 N+1 查询问题的识别
假设某问卷有 1000 份答卷：
- **低效做法：** 遍历 1000 份答卷，每份都执行 `db.users.find_one({"_id": respondent_id})`，总共 1001 次数据库查询（1 次查答卷 + 1000 次查用户）。
- **后果：** 统计接口响应时间线性增长，数据库连接池压力剧增，在高并发场景下可能导致连接耗尽。

### 8.2 批量查询优化方案
`statistics_service.py` 中的 `get_survey_statistics()` 采用了两阶段查询策略：

**阶段一：收集所有需要查询的用户 ID**
```python
respondent_ids = list({
    doc.get("respondent_id")
    for doc in all_responses
    if doc.get("respondent_id") and not doc.get("is_anonymous", True)
})
```
- 使用集合（set）去重，避免同一用户多次提交时重复查询。
- 过滤掉匿名答卷（`is_anonymous=True`），减少无效查询。
- 过滤掉 `respondent_id` 为 `None` 的记录，避免查询失败。

**阶段二：单次批量查询所有用户**
```python
user_map: Dict[str, str] = {}
if respondent_ids:
    users = db.users.find({"_id": {"$in": respondent_ids}})
    for user in users:
        user_map[str(user["_id"])] = user.get("username", "未知用户")
```
- 使用 `$in` 操作符一次性查询所有用户，将 1000 次查询压缩为 2 次（1 次查答卷 + 1 次批量查用户）。
- 构建 `user_id -> username` 的内存映射表（`user_map`），后续统计时直接查表，O(1) 时间复杂度。

### 8.3 性能提升量化
- **查询次数：** 从 O(n) 降低到 O(1)（n 为答卷数）。
- **响应时间：** 在 1000 份答卷的场景下，从约 10 秒降低到约 0.5 秒（假设单次查询 10ms）。
- **数据库负载：** 大幅减少数据库连接占用和查询压力。

### 8.4 匿名用户去重的特殊处理
在选择题的 `respondents` 列表中，需要对答题者去重（同一用户多次选择同一选项时只显示一次）：

**实名用户去重：**
```python
dedup_key = f"real:{respondent['respondent_id']}"
```
- 使用 `respondent_id` 作为去重键，同一用户的多次提交会被合并。

**匿名用户去重：**
```python
dedup_key = f"anon:{entry.get('response_id')}"
```
- 因为匿名答卷的 `respondent_id` 被强制置为 `None`（隐私保护），无法用于去重。
- 改用 `response_id` 作为去重键，确保每份答卷独立展示（即使是同一用户的多次匿名提交，也会显示为多个"匿名用户"）。
- 这种设计在保护隐私的同时，保证了统计数据的完整性。

### 8.5 单次遍历构建所有题目统计
```python
for q in questions:
    stat = _build_question_statistic(q, all_responses, user_map)
    question_statistics.append(stat)
```
- 虽然外层循环遍历题目列表，但 `all_responses` 只查询一次，所有题目共享同一份答卷数据。
- `_build_question_statistic()` 内部会过滤出该题的答案（`if ans.get("question_id") == qid`），避免重复查询数据库。
- 时间复杂度：O(题目数 × 答卷数)，但只有一次数据库查询，整体性能可接受。

## 9. 历史数据容错的具体实现细节

当问卷在 `closed` 状态下被修改（删除题目、删除选项）后，历史答卷中可能存在"孤儿数据"（引用了已不存在的 `question_id` 或 `option_id`）。

### 9.1 孤儿选项检测机制
`statistics_service.py` 的 `_build_question_statistic()` 在统计选择题时会检测孤儿选项：
```python
for entry in answer_entries:
    answer = entry.get("answer")
    if q_type == "single_choice" and isinstance(answer, str):
        if answer in option_counts:
            option_counts[answer] += 1
        else:
            # 历史答卷中的选项在当前题目定义中已不存在
            unknown_option_count += 1
```
- 当答案中的 `option_id` 不在当前题目的 `options` 列表中时，计入 `unknown_option_count`。
- 这些孤儿答案不会导致程序崩溃，而是被静默跳过。

### 9.2 警告信息的生成与传递
检测到孤儿数据后，统计结果中会添加警告字段：
```python
if unknown_option_count > 0:
    result["warning"] = f"有 {unknown_option_count} 个答案的选项在当前题目定义中已不存在（可能是历史数据）"
```
- 前端可以在统计页面显示此警告，提醒问卷创建者数据不完整的原因。
- 不影响正常选项的统计结果，只是提供额外的诊断信息。

### 9.3 为什么选择"静默跳过"而非"抛出异常"
**设计权衡：**
- **抛出异常的问题：** 如果遇到孤儿数据就返回 500 错误，会导致整个统计功能不可用，问卷创建者无法查看任何统计结果，用户体验极差。
- **静默跳过的优势：** 
  - 保证统计功能的可用性，至少能看到有效数据的统计。
  - 通过警告信息告知问题，而非直接拒绝服务。
  - 符合"优雅降级"的工程实践，在数据不完美的情况下仍提供部分功能。

**适用场景：**
- 问卷创建者在 `closed` 状态下修改了题目结构（删除选项、删除题目）。
- 历史答卷中引用的 `question_id` 或 `option_id` 在当前问卷定义中已不存在。
- 统计时自动忽略这些孤儿数据，并在结果中添加 `warning` 字段提示。

### 9.4 孤儿题目的容错处理
除了孤儿选项，统计引擎还处理了孤儿题目的情况：
```python
for q in questions:
    stat = _build_question_statistic(q, all_responses, user_map)
    question_statistics.append(stat)
```
- 统计时只遍历当前问卷定义中的题目列表。
- 如果历史答卷中包含已被删除题目的答案，这些答案不会出现在统计结果中（因为题目本身已不在 `questions` 数组中）。
- 这种设计确保统计结果始终与当前问卷结构保持一致，避免展示"幽灵题目"的统计数据。
