# 第二阶段数据库设计文档

## 设计总览

第二阶段在第一阶段三集合（`users`、`surveys`、`responses`）基础上，新增 `questions` 集合，实现题目独立管理、版本控制、共享复用与跨问卷统计。

**核心设计变更：**

- 新增 `questions` 集合：独立存储题目，支持版本管理、共享与题库功能
- `surveys` 集合的 `questions` 数组从"嵌入完整题目内容"改为"引用题目 ID + 版本号"
- `responses` 集合的 `answers` 数组新增 `q_id` 和 `version_number`，支持跨问卷统计

**核心设计决策：**

| 决策点 | 结论 | 理由 |
|--------|------|------|
| 是否需要独立「题库」collection？ | ❌ 不需要 | 需求中的"题库管理"本质是用户的个人收藏夹（加入/移除/选题），通过 `banked_by` 数组即可实现，不涉及多分组管理 |
| survey 中是否嵌入 questions 内容？ | ❌ 不嵌入，使用引用 | 前提是"旧版本不能删、不能改、始终可查"。引用方案使跨问卷统计（需求八）和反查引用（需求六）成为简单查询，仅多 1 次批量查询的代价 |
| 版本存储模式 | 内嵌版本模式 | 版本与题目强绑定，一次查询获取全部历史，权限统一管理 |
| 共享模式 | 共享同一实体 | 保持 `q_id` 不变，支持跨问卷统计和题库动态更新 |
| 题目删除策略 | 不实现删除 | 需求未要求删除题目，且"保留一切"的精神与版本不可变原则一致；仅支持"从题库移除"（`$pull banked_by`） |

## 集合（Collections）设计

### 1. questions（题目）集合 —— 新增

独立存储题目实体，支持版本管理、共享与题库功能。

```json
{
  "_id": ObjectId,                    // MongoDB 自动主键
  "q_id": String,                     // 题目业务唯一标识（如 "q_1002"），表示同一题目的版本谱系
  "latest_version_number": Number,    // 最新版本号，通过 $inc 原子递增保证并发安全

  // 权限与状态
  "access_control": {
    "creator": String,                // 题目最初的创建者（user_id），拥有完全控制权
    "shared_with": [String],          // 共享给哪些用户（user_id 数组），只读使用权（可引用、可加入题库，不能创建新版本）
    "banked_by": [String]             // 哪些用户把它加入了自己的题库（user_id 数组）
  },

  // 版本与内容
  "versions": [
    {
      "version_number": Number,       // 版本编号，从 1 开始递增
      "created_at": DateTime,         // 该版本创建时间
      "parent_version_number": null,  // 父版本号（第一个版本为 null，恢复旧版本时指向源版本）

      // 该版本的题目内容
      "type": String,                 // 题目类型："single_choice" / "multiple_choice" / "text_input" / "number_input"
      "title": String,                // 题目文本
      "required": Boolean,            // 是否必答
      "options": [                    // 选项（仅 single_choice / multiple_choice 使用）
        {
          "option_id": String,        // 选项标识
          "text": String              // 选项文本
        }
      ],
      "validation": {                 // 校验规则（按题型区分，与第一阶段一致）
        // 多选题：min_selected, max_selected, exact_selected
        // 文本题：min_length, max_length
        // 数字题：min_value, max_value, integer_only
      }
    }
  ]
}
```

**索引：**

- `q_id`（唯一索引，题目业务标识定位）
- `access_control.creator`（查询"我创建的题目"）
- `access_control.shared_with`（多键索引，查询"共享给我的题目"）
- `access_control.banked_by`（多键索引，查询"我的题库"）

**设计说明：**

关于共享：

- 共享的是**同一个题目实体**，而不是复制一份新题目（保持 `q_id` 不变）。
  - **为了实现跨问卷统计（需求八）：** 团队数据汇总统计，实现"这道题在全团队所有问卷中的总体数据"的聚合计算。
  - **为了题库的动态更新（统一标准）：** 如果你分享了 v1，后来更新到了 v2，同事打开"共享空间"时能看到最新版本并选择使用。
- `creator` 拥有完全控制权（创建新版本、管理共享），`shared_with` 用户仅有**只读使用权**（引用到自己的问卷、加入自己的题库）。

关于版本：

- 使用**内嵌版本模式（Embedded Versioning Pattern）**，每次修改题目向 `versions` 数组 `$push` 新对象，版本号通过 `$inc` 原子递增。
- 通过 `parent_version_number` 可回溯版本来源，支持 v1 → v2 → v3 的版本关系展示（可能有分支）。
- 恢复旧版本时，不直接覆盖历史版本，而是基于目标旧版本内容创建新版本（`parent_version_number` 指向旧版本）。
- 版本一旦写入**不可修改、不可删除**，保证已引用该版本的问卷数据完整性。
- 权限挂在最外层，一次查询获取全部历史，无需额外查表。

关于并发安全：

```python
# 创建新版本时使用原子操作，保证 version_number 不冲突
result = db.questions.find_one_and_update(
    {"q_id": "q_1002"},
    {
        "$inc": {"latest_version_number": 1},
        "$push": {"versions": new_version_obj}
    },
    return_document=ReturnDocument.AFTER
)
```

关于题库（不需要独立 collection）：

- "题库管理"本质是用户的个人收藏夹：加入题库 → `$addToSet banked_by`，移除 → `$pull banked_by`，查询 → `find({"access_control.banked_by": user_id})`。
- 当前需求不涉及"创建多个命名分类题库"，`banked_by` 方案完全满足。

关于文档大小：

- MongoDB 单文档 16MB 限制。对于文本为主的问卷题目，即使存 1000 个版本也不会超限，当前业务下完全安全。

### 2. surveys（问卷）集合 —— questions 部分修改为引用

问卷的 `questions` 数组从嵌入完整题目内容改为引用 `q_id + version_number`，仅保留问卷局部上下文。

```json
{
  "_id": ObjectId,
  "title": String,
  "description": String,
  "creator_id": ObjectId,
  "access_code": String,
  "status": String,              // "draft" / "published" / "closed"
  "created_at": DateTime,
  "updated_at": DateTime,
  "deadline": DateTime,
  "response_count": Number,
  "settings": {
    "allow_anonymous": Boolean,
    "allow_multiple": Boolean
  },

  "questions": [
    {
      // 局部上下文（仅本问卷有效，不污染全局题库）
      "question_id": String,     // 问卷内部局部题号（如 "q1"），给 logic 跳转路由用
      "order": Number,           // 题目在此问卷中的顺序
      "logic": {                 // 跳转逻辑（只属于该份问卷）
        "enabled": Boolean,
        "rules": [
          {
            "condition": {
              "type": String,    // "select_option" / "contains_option" / "number_compare"
              // ... 条件详情（与第一阶段一致）
            },
            "action": {
              "type": String,    // "jump_to" / "end_survey"
              "target_question_id": String
            }
          }
        ]
      },

      // 题目溯源（用 q_id + version_number 唯一确定题目内容）
      "q_id": String,            // 题目 ID，引用 questions 集合
      "version_number": Number   // 引用的具体版本号
    }
  ]
}
```

**索引（新增）：**

- `questions.q_id`（多键索引，反查"哪些问卷使用了这道题"——需求六）
- 其余索引与第一阶段保持不变：`creator_id`、`access_code`（唯一）、`status` + `created_at`

**设计说明：**

关于引用 vs 嵌入的选择：

| 维度 | 嵌入（第一阶段） | 引用 q_id + version_number（第二阶段） |
|------|------|------|
| 查询问卷+题目 | 1 次查询 | 2 次查询（先查 survey，再 `$in` 批量查 questions） |
| 跨问卷统计（需求八） | 需扫描所有 survey | 直接按 `responses.answers.q_id` 聚合 |
| 反查引用（需求六） | 需扫描所有 survey | 查 `surveys.questions.q_id` 索引 |
| 题目共享（需求二） | 每个 survey 一份副本 | 共享同一实体，零冗余 |
| 数据一致性 | 自包含 | 依赖不变量（旧版本不删不改） |

引用方案仅多 1 次批量查询，但换来跨问卷统计、题目追踪、共享等功能的巨大便利。

关于题目修改：

- 每次保存的新修改都会自动作为题目的新 version 保存；已使用旧版本的其他问卷保持原引用版本不变。
- 已发布状态（`published`）的问卷题目不能改，草稿（`draft`）和关闭（`closed`）状态可以。
- 如果发布后又关闭，旧版本题目已收集到回复后进行了修改，统计结果需按版本分别展示，不能统一覆盖（配合 `responses` 中的版本绑定实现）。

关于 `question_id`、`order` 和 `q_id` 三个字段的说明：

- `question_id`：问卷内部局部题号（如 "q1"），给跳转逻辑路由用。
- `q_id`：题目实体的全局唯一 ID，标识"这是同一个题目的不同版本"。
- `order`：题目在此问卷中的显示顺序。

为什么三者缺一不可：

- **无法用 `q_id` 代替 `question_id`**：用户可能将同一题库模板（如 "1-5 星满意度打分" `q_1002`）添加两次，分别评价"服务态度"和"专业能力"，`q_id` 相同会导致跳转错误。
- **无法用 `order` 代替 `question_id`**：拖拽调整顺序后 `order` 变化，所有跳转逻辑都需重写，不合理。

### 3. responses（答卷）集合 —— 新增版本绑定

答卷的 `answers` 数组新增 `q_id` 和 `version_number`，支持跨问卷统计和版本追踪。

```json
{
  "_id": ObjectId,
  "survey_id": ObjectId,
  "respondent_id": ObjectId,     // 匿名时为 null
  "is_anonymous": Boolean,
  "submitted_at": DateTime,
  "answers": [
    {
      "question_id": String,     // 问卷内部局部题号
      "q_id": String,            // 新增：题目全局 ID，方便跨问卷统计
      "version_number": Number,  // 新增：记录回答对应的题目版本号
      "answer": Mixed            // 答案值（与第一阶段格式一致）
                                 // - single_choice: String（option_id）
                                 // - multiple_choice: [String]（option_id 数组）
                                 // - text_input: String
                                 // - number_input: Number
    }
  ],
  "completion_time": Number
}
```

**索引（新增）：**

- `answers.q_id`（多键索引，跨问卷统计——需求八）
- 其余索引与第一阶段保持不变：`survey_id` + `submitted_at`、`survey_id` + `respondent_id`、`respondent_id`

**设计说明：**

- `answer` 保持第一阶段的 `Mixed` 类型统一格式，不新增冗余字段。
- `q_id` + `version_number` 使跨问卷统计成为可能：按 `q_id` 聚合所有问卷的回答，按 `version_number` 分组可观察不同版本的数据差异。

## 需求覆盖验证

| 需求 | 如何满足 |
|------|----------|
| 一、保存常用题目 | `questions` 集合独立存储，survey 通过 `q_id + version_number` 引用 |
| 二、分享题目 | `access_control.shared_with` 数组，共享同一实体 |
| 三、修改不影响旧问卷 | 版本化：每次修改生成新版本，旧问卷保持原版本引用不变 |
| 四、记录修改历史 | `versions` 内嵌数组 + `parent_version_number` 追溯来源 |
| 五、不同版本同时存在 | 内嵌版本数组天然支持，不同问卷可引用不同版本号 |
| 六、查看题目被哪些问卷使用 | 查询 `surveys.questions.q_id`（多键索引） |
| 七、建立题库 | `access_control.banked_by` 数组实现用户个人题库 |
| 八、跨问卷统计 | `responses.answers.q_id` 聚合，按 `version_number` 可分版本统计 |
