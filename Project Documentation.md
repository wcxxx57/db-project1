# **项目：基于** **MongoDB** **的问卷系统开发**

**一、项目目标**

本项目要求开发一个简化版的在线问卷系统，功能类似于“问卷星”或 Google Forms。

系统应允许注册用户创建问卷，并通过URL分发问卷，其他用户可以填写问卷，系统可以统计问卷结果。

本项目重点考察：

- MongoDB     数据库设计能力     
- 后端程序设计能力 
- 数据结构与业务逻辑建模能力 
- 条件逻辑与跳转逻辑的实现能力 
- 使用 AI 编程工具的能力与规范     

本项目**不要求复杂界面**，可以使用简单网页。

评分重点：

数据库设计 + 后端逻辑 + 文档质量 + AI使用过程说明



**二、项目说明**

本课程项目将模拟真实互联网应用开发的过程。
 在实际的软件开发中，用户需求往往不是一次性确定的，而是会随着使用过程不断增加和变化。开发者通常需要在已有系统的基础上进行修改和扩展，而不是每次都重新设计和实现整个系统。

为了让大家体验这种真实的软件开发过程，本项目分为 **两个阶段** 完成。



**第一阶段：根据初始需求完成系统设计与实现**

在第一阶段会提供一份**初始用户需求说明**。

同学需要根据这份需求：

- 进行系统设计 
- 完成数据库设计 
- 实现前后端程序 
- 完成必要的测试 
- 编写项目文档 
- 提交完整的项目报告 

**整个过程尽可能利用AI编程工具完成。**

第一阶段提交的内容参见“七、提交内容”。

第一阶段的成果将作为第二阶段的基础。



**第二阶段：根据用户需求变更对系统进行修改与扩展**

在第一阶段提交完成之后，教师会发布一份**新的用户需求变更说明**。

该需求变更模拟真实应用中的情况，例如：

- 用户提出新的功能要求 
- 原有功能需要扩展 
- 原有设计无法满足新的使用方式 
- 数据需要以新的方式组织和统计 

在第二阶段，同学需要：

- 在第一阶段系统的基础上进行修改和扩展     
- 根据新的需求调整数据库设计 
- 修改程序实现 
- 保证旧功能仍然可用（除非需求明确要求改变）     
- 完成新的测试 
- 再次提交完整的项目报告 

**整个过程也尽可能利用AI编程工具完成。**



**重要要求：不得推倒重做**

第二阶段 **不允许丢弃第一阶段的成果重新实现**。

必须满足以下要求：

- 必须在第一阶段代码基础上修改 
- 不得重新设计一个完全新的系统替代原系统     
- 不得删除第一阶段的主要功能 
- 必须保留第一阶段的核心结构，并进行合理扩展     

在最终报告中必须说明：

- 第一阶段的设计是什么 
- 新需求带来了哪些问题 
- 为什么需要修改原设计 
- 修改了哪些部分 
- 数据结构如何变化 
- 程序如何调整 
- 是否遇到兼容性问题 
- 如何保证旧功能仍然可用 

即需要清楚记录整个修改过程。



**为什么分成两个阶段**

本项目的目标不是实现一个功能最多的系统，而是让同学体验：

- 用户需求不确定 
- 需求会不断变化 
- 数据结构需要调整 
- 系统需要迭代升级 
- 旧数据必须保留 
- 旧功能不能随意删除 

这些都是实际互联网应用开发中的常见情况。

通过两个阶段的开发，希望同学能够理解：

软件系统不是一次设计完成的，而是在不断变化的需求中逐步演化的。



**三、技术要求**

**初始用户需求**

**一、用户账号功能**

系统需要支持用户注册和登录。

用户希望能够：

- 注册账号 
- 登录账号 
- 使用自己的账号创建问卷 
- 查看自己创建的问卷 
- 查看问卷的统计结果 

每个用户至少需要有：

- 用户名 
- 密码 
- 注册时间 

用户登录后，应只能看到自己创建的问卷。



**二、问卷创建与管理**

用户希望能够创建问卷。

创建问卷时可以填写：

- 问卷标题 
- 问卷说明 
- 是否允许匿名填写 

用户希望能够：

- 查看自己创建的问卷列表 
- 发布问卷 
- 设置问卷截止时间
- 关闭问卷 

每个问卷都应该有一个可以访问的链接，例如：

/survey/xxxxxx

用户可以把这个链接发给别人，让别人填写问卷。

填写问卷的人需要先登录。



**三、问卷中的题目**

一个问卷可以包含多个题目。

用户希望在问卷中添加不同类型的问题。

系统至少需要支持以下三种题型。



**1.** **单选题**

示例：

你的性别

- 男 
- 女 
- 其他 

用户希望能够：

- 设置选项 
- 设置是否必须回答 

如果是必答题，填写问卷时必须选择。



**2.** **多选题**

示例：

你喜欢的水果

  苹果 

  香蕉 

  西瓜 

  葡萄 

用户希望可以限制选择数量，例如：

- 至少选择 2 个     
- 最多选择 3 个     
- 必须选择 2 个     

系统在填写时需要检查是否符合要求。



**3.** **填空题**

填空题可以是文字或数字。

用户希望可以创建：

- 文本填空 
- 数字填空 

对于文本填空，希望可以限制：

- 最少输入多少字 
- 最多输入多少字 

对于数字填空，希望可以限制：

- 最小值 
- 最大值 
- 是否必须是整数 

例如：

  年龄：必须为整数，且在0~120之间

  手机号：必须为整数

填写时系统应检查是否合法。



**四、题目跳转功能**

用户希望问卷可以根据回答情况跳转到不同题目。

例如：

如果第1题选 A → 跳到第5题
 如果第1题选 B → 跳到第3题

用户希望可以设置：

- 根据单选结果跳转 
- 根据多选结果跳转 
- 根据数字填空内容跳转 

填写问卷时，系统应自动跳到对应题目。

用户不希望跳转写死在程序里，而是可以在问卷中设置。



**五、填写问卷**

填写问卷的人可以通过问卷链接进入问卷。

填写时希望：

- 按顺序回答问题 
- 根据规则跳转 
- 如果填写不符合要求，系统提示错误 
- 填写完成后可以提交 

每次填写都应该被保存。

系统需要支持：

- 多人同时填写 
- 匿名填写 
- 同一个人填写多次（如果允许） 



**六、查看统计结果**

创建问卷的人希望可以查看统计结果。

统计结果至少包括：

**单选题**

- 每个选项有多少人选择 
- 总共有多少人回答 

**多选题**

- 每个选项被选择多少次 

**填空题**

- 可以查看所有填写内容 
- 如果是数字，可以查看平均值 

用户希望能够：

- 查看整个问卷的统计 
- 查看某一道题的统计 

统计结果应该根据已经填写的数据自动生成。
